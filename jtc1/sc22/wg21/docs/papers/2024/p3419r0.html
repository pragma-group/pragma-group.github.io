1 Goals

This paper presents a summary of reflection syntax options. Some of these options are activately under consideration, while others have already been discussed and dismissed. However, the reasons for their dismissal have not necessary been recorded in a central location showcasing their perceived advantages or disadvantages relative to other options. This paper intends to be a reference for such discussions. A specific non-goal of the paper is the promotion of a particular syntax.

3.2 Advantages and Disadvantages

# 1: ^^foo

  • Advantages
    1. There is no ambiguity with blocks [P3381R0].
    2. This is the syntax recommended by [P3381R0].
  • Disadvantages
    1. It is longer than the status quo [P3381R0].
    2. It could potentially be used for logical xor.
    3. Its use as a binary operator has a syntactic ambiguity with Clang’s blocks extension and will probably not be able to be used for exclusive-or. An example is expr ^^ { return 1; }().

# 2: ^foo

  • Advantages
    1. The upward arrow style notation indicates something is being pulled from foo.
    2. It is a relatively rarely used binary operator [P3381R0].
  • Disadvantages
    1. Clang has an old extension called “blocks” that uses ^ to both introduce a block and as part of the syntax for “block types” [P3381R0]. With this extension, some parses may not be able to be disambiguated. type-id(^ident)() is ambiguous, and auto A = ^{ f(); }; with [P3294R1] is ambiguous [P3381R0].
    2. It is an operator being used in C++ for non-operator syntax.
    3. It is too similar to foo^, which is used as a type in other languages built on top of C++ such as ObjectiveC++ [P3381R0] and the Microsoft .NET extensions.

# 3: @<foo

  • Advantages
    1. It is similar to Go’s channel syntax [GoChannels].
    2. It shouldn’t have a nesting issue as the character order solves it.
  • Disadvantages
    1. There is potential concern about readability with the angle brackets in a template argument context, an example of which would be fn<@<foo>.

# 4: @<<foo

  • Advantages
    1. It makes the direction louder and easier to read than using @<foo.

# 5: @<-foo

  • Advantages
    1. It further emphasizes the direction of the data flow.
  • Disadvantages
    1. It might need parentheses for the - sign.

# 7: |foo

  • Advantages
    1. It is not part of a digraph [P3381R0].
    2. This is not ambiguous to compilers [P3381R0].
  • Disadvantages
    1. It looks like half of an absolute value notation and could be confused as part of a magnitude notation when using multiple reflections in a single line as in if (|T == |U) [P3381R0].

# 8: /foo

  • Advantages
    1. It can offer nice symmetry with using \ for code injection [P3381R0].
  • Disadvantages
    1. It is very close to comment syntax and could require parentheses for /(/e) and /(*e) [P3381R0].

# 9: %foo

  • Advantages
    1. It is a relatively rarely used binary operator [P3381R0].
  • Disadvantages
    1. When passed as a first template argument as in C<%T> c;, <% is a digraph for {. This would require spaces or parentheses to get around this issue [P3381R0].

# 10: `foo

  • Advantages
    1. This is a very lispy syntax. In Common Lisp it is used to disable evaluation for a subexpression [CLSpec].
    2. It was recently added to the basic character set [P3381R0].
    3. It is a small character, even smaller than ^ [P3381R0].
  • Disadvantages
    1. Markdown uses it extensively, and not all Markdown implementations give you ways to escape it [P3381R0].

# 11: :foo

  • Advantages
    1. It is used by the Julia programming language for metaprogramming to construct inline expressions such as ex2 = Expr(:call, :+, 1 , 1) to produce a Julia Expr type such as :(1 + 1) [JuliaMeta].
  • Disadvantages
    1. Passing it as a template argument as in C<:T> would not work because <: is a digraph for [ [P3381R0]. This can potentially be remedied by being called as C<(:T)>, but this could leads to errors that are difficult to debug.
    2. It composes poorly with the splicing operator [::foo:].
    3. It may be ambiguous when parsing an array-index whose operand is a qualified-id or when parsing a reflect-expression composed inside a splice-specifier.

# 12: ~~foo

  • Disadvantages
    1. It is ambiguous. If foo is an integral expression, then ~~foo means to flip all of the bits and then flip them again.
    2. It may require parentheses to parse the intended, single ~ symbol.
    3. ~ is already a unary operator.

# 13: ^-foo

  • Disadvantages
    1. It may require parentheses to parse the - symbol.

# 14: <-foo

  • Advantages
    1. It is similar to Go’s channel syntax [GoChannels].
  • Disadvantages
    1. It could be hard to disambiguate <<- from < <- when used as a template argument as in fn<<-foo>.
    2. It may require parentheses to parse the - symbol.

# 18: &&foo

  • Disadvantages
    1. It can lead to code like this: if (&&foo&& == &&foo&& && &&foo&& == &&foo&&) where &&foo&& is a reflection on a foo reflection type. This is comparing two reflections for equality.
    2. It can be an issue with reflecting on an rvalue reference as in (&&foo&&).
    3. The && is already used for logical AND and rvalue/forward references.

# 19: @foo

  • Disadvantages
    1. It was potentially going to be used as a splat for simple splices.
    2. Compilers already allow the use of @ in identifiers as an extension, so we wouldn’t be able to disambiguate between reflecting on an identifier and actually being an identifier [P3381R0].
    3. It is used as an extension in Objective-C. As such, a number of potential identifiers have meaning already (@property, @dynamic, @optional, etc). This also cannot be escaped since @(foo) is a boxed expression and @[foo] and @{foo} are container literals [P3381R0].

# 20: #foo

  • Disadvantages
    1. It precludes usage in existing C macros, given that #define MACRO(x) do_something_with(#x) already has meaning [P3381R0].

# 21: $foo

  • Advantages
    1. It doesn’t conflict with anything else in C++ since it was recently added to the basic character set [P3381R0].
  • Disadvantages
    1. Compilers support its use in identifiers as an extension [P3381R0].
    2. It is typically used as a splicing syntax in other languages, not as the get meta syntax.

# 23: =foo

  • Advantages
    1. It is already used as a binary operator [P3381R0].
  • Disadvantages
    1. It is too commonly used as a binary operator to now overload it for reflection usage [P3381R0].

# 24: ?foo

  • Advantages
    1. It already exists within the language today [P3381R0].
    2. It would not be ambiguous with the conditional operator [P3381R0].
    3. The syntax naturally implies asking what an entity is, and the answer is its reflection [P3381R0].
  • Disadvantages
    1. It is associated with predicates in different programming languages for null coalescing, optional chaining, and error propagation (v.empty?) [P3381R0].
    2. [P2688R1] proposes using it for pattern matching [P3381R0].

# 25: \foo

  • Advantages
    1. It is an available operator.
  • Disadvantages
    1. There is an issue with UCNs. \u0 is parsed as a UCN instead of a reflection of u0 [P3381R0].
    2. Some prefer it as an interpolator, for which there is prior art [P3381R0].

# 26: ^[foo]

  • Advantages
    1. It doesn’t have an issue with block ambiguity, as opposed to ^foo [P3381R0].
    2. It can be more symmetric with splicing if splicing uses similar syntax like [: foo :] [P3381R0].
  • Disadvantages
    1. It is a heavier syntax [P3381R0].

# 27: ${foo}

  • Advantages
    1. It does not conflict with compiler support for using $ in identifiers as an extension [P3381R0].
  • Disadvantages
    1. The $ symbol is typically used as a splicing syntax in other languages, not as the get meta syntax.
    2. There might be an issue with token sequences that might require an extra set of braces [P3381R0].

# 28: $(foo)

  • Advantages
    1. It does not conflict with compiler support for using $ in identifiers as an extension [P3381R0].
  • Disadvantages
    1. The $ is typically used as a splicing syntax in other languages, not as the get meta syntax.

# 29: /\foo

  • Advantages
    1. It is the big caret equivalent of the ^ symbol.
  • Disadvantages
    1. There is an issue with UCNs. /\u0 will have \u0 parsed as a UCN instead of a reflection of u0 [P3381R0].

# 30: ``foo

  • Advantages
    1. This is a very lispy syntax. In Common Lisp the single `foo is used to disable evaluation for a subexpression [CLSpec].
  • Disadvantages
    1. Markdown uses three backquotes to indicate the start and end of a code block, so it might be hard for Markdown to distinguish between starting a code block and capturing an inline code block of a reflection of foo.

4.2 Advantages and Disadvantages

# 6: (: foo :)

  • Disadvantages
    1. It might be confusing to someone reading the code quickly.
    2. It will render as a smiley face.

# 9: ->foo

  • Advantages
    1. It is similar to Go’s channel syntax [GoChannels].
  • Disadvantages
    1. It could be a good relocation syntax in C++29.
    2. It could be difficult to parse, particularly as a template argument as in fn<->foo>.

# 10: >-foo

  • Disadvantages
    1. There is a potential parsing ambiguity. tvar<>-foo>-3 can be parsed as (tvar<> - foo) > -3 or as tvar<(>-foo)> - 3.
    2. It might require parentheses to parse the - symbol for code such as >-(-foo).

# 11: $foo

  • Advantages
    1. This syntax is used for splicing in a number of programming languages.
    2. It doesn’t conflict with anything else in C++ since it was recently added to the basic character set [P3381R0].
  • Disadvantages
    1. Compilers allow using $ in identifiers as an extension, so it can be hard to differentiate between an identifier and reflecting on an identifier [P3381R0].

# 13: ,foo

  • Advantages
    1. In Common Lisp it is used to evaluate and splice in an expression [CLSpec].
  • Disadvantages
    1. It may be difficult to parse.

# 14: ,,foo

  • Disadvantages
    1. It may be difficult to parse. An example could be fn(,,bool, ,,int, ,,char).

# 17: [| foo |]

  • Advantages
    1. It is used in Template Haskell for expression quotation [thQuotes].

# 18 ${foo}

  • Advantages
    1. It does not conflict with compiler support for using $ in identifiers as an extension [P3381R0].
  • Disadvantages
    1. There might be an issue with token sequences that might require an extra set of braces [P3381R0].

# 19 $(foo)

  • Advantages
    1. It does not conflict with compiler support for using $ in identifiers as an extension [P3381R0].

# 20: ,@foo

  • Advantages
    1. In Common Lisp it is used to evaluate and splice in a list of values resulting from an expression [CLSpec].

7.2 Advantages and Disadvantages

# 1: Unary prefix operators

  • Disadvantages
    1. It requires operator precedence rules to parse the code.
    2. It may require inserting parentheses to ensure operational order, similar to pointer dereferencing: *(foo()).

# 2: Bounded splice operators

  • Advantages
    1. They can be easier to parse for nested name specifiers like [:Cls::reflection_member:]::inner::member, which is easier to parse than $Cls::reflection_member::inner::member.

# 3: Single character splice operators

  • Advantages
    1. They are a terser syntax.
  • Disadvantages
    1. There is potentially an issue with postfix operators when it comes to parsing.
    2. It can be difficult to parse complicated expressions that need to be spliced in.

# 4: Function keywords

  • Disadvantages
    1. It is a heavier syntax, and it does not contribute more than a terser syntax to readability [P3381R0].
    2. A reflection is typically taken to pass the reflection to a function that operates on it, not just to get the reflection. As a result, making a reflection syntax stand out is less impactful given its pass-through nature [P3381R0].
    3. Novices will rarely use reflection, so choosing a novice-friendly keyword may not serve its most common user base [P3381R0].
    4. For splicing, unparenthesized keywords, when used as a part of an unqualified-id, make it difficult to know where the splice ends. An example is splice foo::bar::baz.

# 5: Contextual keywords

  • Disadvantages
    1. It is a heavier syntax, and it does not contribute more than a terser syntax to readability [P3381R0].
    2. A reflection is typically taken to pass the reflection to a function that operates on it, not just to get the reflection. As a result, making a reflection syntax stand out is less impactful given its pass-through nature [P3381R0].
    3. Novices will rarely use reflection, so choosing a novice-friendly keyword may not serve its most common user base [P3381R0].
